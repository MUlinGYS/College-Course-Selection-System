# 压力测试结果报告

## 1. 测试范围

- 测试日期：2026-03-20
- 项目名称：校园选课系统
- 测试接口：`POST /api/enrollments`
- 测试目标：验证接入 Redis 锁和数据库优化后的并发选课承载能力
- 服务启动方式：`Waitress`
- 服务地址：`http://127.0.0.1:8000`

## 2. 当前并发保护方案

当前选课接口采用两层保护：

1. Redis 分布式锁
   - 锁维度 1：`student + round`
   - 锁维度 2：`section`
2. MySQL 事务保护
   - 事务块控制
   - `select_for_update` 行级锁

本轮测试前已完成的优化项如下：

- 后端已接入 Redis，用于选课并发保护
- `ENROLLMENT_LOCK_WAIT_TIMEOUT` 由 `1.5s` 调整为 `3.0s`
- 为选课、开课班级、轮次相关查询补充数据库索引
- 服务启动方式由 Django `runserver` 改为 `Waitress`

## 4. 优化前基线结果

在早期使用 Django `runserver` 进行验证时，得到的基线情况如下：

| 启动方式 | 并发量 | 结果 |
| --- | --- | --- |
| `runserver` | 100 | 基本稳定 |
| `runserver` | 150 到 200 | 性能退化明显，不适合作为正式压测结论 |

说明：这一轮只保留了结论性结果，没有完整保留详细延迟数据，主要用于体现优化前后差异。

## 5. 优化后压力测试结果

完成 `Waitress + Redis 锁 + 索引优化` 后，压测结果如下：

| 并发量 | 成功率 | P95 延迟 | 说明 |
| --- | --- | --- | --- |
| 100 | 100% | 1480.81 ms | 稳定 |
| 150 | 100% | 2144.96 ms | 稳定 |
| 200 | 100% | 2419.60 ms | 稳定 |
| 250 | 100% | 未单独保留 | 稳定 |
| 300 | 100% | 未单独保留 | 稳定 |
| 400 | 99.5% | 未单独保留 | 仅出现 2 次 `REQUEST_BUSY` |

## 6. 结果分析

### 6.1 稳定性结论

- 系统目前可以稳定支撑 `200 到 300` 并发选课请求
- 在 `400` 并发时已经接近上限，但仍保持 `99.5%` 成功率
- 高并发下的失败类型主要是受控拒绝，例如 `REQUEST_BUSY`，而不是数据错乱

### 6.2 业务层意义

当前并发保护方案已经能够有效避免以下风险：

- 同一学生重复提交选课请求
- 班级容量被并发超卖
- 高并发下的竞争写入问题
- 并发请求导致的数据不一致问题

### 6.3 部署层意义

- `runserver` 只适合开发调试
- 正式演示、联调、压测、部署应使用 `Waitress`
- `Redis + MySQL 行级保护` 已经能够支撑本次毕设范围内的高并发选课场景

## 7. 可直接写入论文的结论

压力测试结果表明，在引入 Redis 分布式锁、MySQL 事务行级锁、数据库索引优化以及 Waitress 服务部署后，校园选课系统的核心选课接口并发处理能力得到明显提升。相较于早期基于 `runserver` 的测试结果，优化后的系统能够在 `200 到 300` 并发下保持 `100%` 成功率，在 `400` 并发下仍可维持 `99.5%` 成功率。说明本文设计的高并发选课控制机制能够有效保证选课过程中的一致性与稳定性，满足当前系统预期使用需求。
